# 자기소개서/경력 기술서 작성 가이드: 'Curve' 프로젝트 활용 전략

이 프로젝트('Curve')는 단순한 CRUD 애플리케이션이 아니라, **아키텍처 설계 능력, 분산 시스템에 대한 이해, 그리고 라이브러리 설계(DX) 능력**을 모두 보여줄 수 있는 매우 강력한 포트폴리오입니다.

이직 시 자기소개서와 면접에서 이 프로젝트를 어떻게 녹여내면 좋을지, **핵심 전략과 구체적인 문구**를 제안합니다.

---

## 1. 핵심 전략: "무엇을 만들었나"보다 "어떤 문제를 해결했나"에 집중

단순히 "Kafka 라이브러리를 만들었습니다"라고 하면 임팩트가 약합니다. 다음과 같은 **문제 해결 스토리**로 접근하세요.

1.  **생산성 문제**: "팀원들이 매번 Kafka 연동 코드를 50줄씩 작성하고 있었다." → **선언적 어노테이션(@PublishEvent)으로 1줄로 단축.**
2.  **데이터 정합성 문제**: "DB 트랜잭션은 성공했는데 메시지 발행이 실패하는 'Dual-Write' 문제가 있었다." → **Transactional Outbox 패턴 구현으로 원자성 보장.**
3.  **유지보수성 문제**: "비즈니스 로직과 인프라 코드가 뒤섞여 있었다." → **Hexagonal Architecture로 Core와 Infra를 완벽히 분리.**
4.  **운영 안정성 문제**: "비동기 처리 시 로그 추적이 끊기고, 장애 시 데이터가 유실되었다." → **MDC Context 전파 및 3단계 장애 복구(Retry -> DLQ -> File Backup) 구현.**

---

## 2. 자기소개서 / 경력 기술서 작성 예시

### A. 프로젝트 소개 (한 줄 요약)

> **[Spring Boot 기반의 선언적 이벤트 발행 라이브러리 'Curve' 개발]**
> *마이크로서비스 환경에서 이벤트 발행의 복잡성을 추상화하고, 데이터 일관성과 운영 안정성을 보장하는 사내 표준 프레임워크 개발*

### B. 주요 성과 및 기술적 도전 (Bullet Points)

*   **Hexagonal Architecture 도입을 통한 유연성 확보**
    *   도메인 로직(Core)을 Spring/Kafka 등 외부 기술로부터 격리하여 순수 Java로 구현.
    *   `EventProducer` 포트(Port) 설계를 통해 메시지 브로커 교체(Kafka ↔ RabbitMQ 등)가 용이한 구조 설계.
*   **분산 환경 데이터 정합성 보장 (Transactional Outbox Pattern)**
    *   DB 트랜잭션과 이벤트 발행의 원자성을 보장하기 위해 Outbox 패턴을 직접 구현.
    *   Polling 방식의 퍼블리셔를 구현하고, **지수 백오프(Exponential Backoff)** 전략을 적용하여 실패 시 DB 부하를 최소화하며 재시도 효율 증대.
    *   다중 인스턴스 환경에서의 중복 발행 방지를 위해 `SKIP LOCKED` 기반의 비관적 락 적용.
*   **운영 관측성(Observability) 및 추적성 강화**
    *   비동기 스레드 및 DLQ 처리 과정에서 Trace ID가 유실되는 문제를 해결하기 위해 **MDC Context 전파 로직** 구현.
    *   Micrometer 기반의 커스텀 메트릭(`curve.events.published` 등)을 노출하여 Prometheus/Grafana 모니터링 환경 구축.
*   **안정적인 장애 대응 메커니즘 구축**
    *   Kafka 장애 시 데이터 유실 제로를 목표로 **3단계 복구 전략(Main Topic → DLQ → Local File Backup)** 설계 및 구현.
    *   로컬 파일 백업 시 POSIX 권한 제어(600) 및 PII(개인정보) 마스킹을 적용하여 보안성 확보.
*   **개발자 경험(DX) 개선**
    *   SpEL(Spring Expression Language)을 지원하여 유연한 페이로드 추출 기능 제공.
    *   `MockEventProducer` 등 테스트 유틸리티를 제공하여 라이브러리 사용자의 테스트 용이성 확보.

## 3. "실무 사용의 한계"를 솔직함과 성장 가능성으로 연결

이 프로젝트가 당장 실무에서 사용되기에는 검증과 시간이 더 필요하다는 점을 인지하고 있음을 보여주는 것은 오히려 **메타인지 능력과 성장 가능성**을 어필하는 기회가 됩니다.

### 자기소개서 문구 예시:

> **[기술적 깊이를 위한 도전: 분산 환경 이벤트 처리 라이브러리 개발]**
>
> 실무에서 마이크로서비스 간 데이터 정합성 문제와 반복되는 이벤트 발행 코드로 인한 생산성 저하를 경험했습니다. 이를 해결하기 위한 아키텍처를 고민하던 중, 단순히 기존 라이브러리를 사용하는 것을 넘어 **직접 핵심 메커니즘을 구현해보며 원리를 깊이 이해하고자** 'Curve'라는 라이브러리를 개발했습니다.
>
> 이 프로젝트는 상용 수준의 완성도보다는 **아키텍처 설계와 핵심 문제 해결 능력 향상**에 초점을 맞췄습니다.
>
> *   **Transactional Outbox 패턴 구현**: DB 트랜잭션과 메시지 발행의 원자성을 보장하기 위해 Outbox 패턴을 직접 구현하며 'Dual-Write' 문제의 본질을 이해했습니다.
> *   **Hexagonal Architecture 적용**: 비즈니스 로직과 인프라 기술을 분리하여 유연한 설계를 시도했습니다.
> *   **운영 관점의 고민**: 비동기 처리 시 로그 추적성(MDC) 확보와 장애 복구 전략(Retry, DLQ)을 구현하며 운영 환경에서의 안정성을 고민했습니다.
>
> **[한계점 및 향후 과제]**
>
> 이 라이브러리는 기술적 원리 탐구와 역량 강화를 위한 프로젝트였기에, 실제 대규모 운영 환경에 즉시 적용하기에는 추가적인 검증과 고도화가 필요함을 인지하고 있습니다. 특히 현재 Outbox 패턴의 Polling 방식은 대규모 트래픽 환경에서 DB 부하 및 지연 시간이 발생할 수 있어, **CDC(Change Data Capture) 기반의 아키텍처로 발전시키는 방향을 고민**하고 있습니다. 이러한 한계점을 명확히 인식하고 있으며, 앞으로 실무에서 이러한 문제들을 해결하며 더 나은 시스템을 구축하는 데 기여하고 싶습니다.

### 면접 예상 질문 및 답변 가이드: "왜 굳이 직접 만들었나?"

> Q: "Spring Cloud Stream이나 Debezium 같은 검증된 솔루션이 있는데, 왜 직접 라이브러리를 개발했나요?"
>
> A: "물론 실무에서는 검증된 프레임워크를 사용하는 것이 맞습니다. 하지만 저는 **도구의 사용법만 익히는 개발자가 아니라, 도구가 해결해주는 문제의 본질을 이해하는 개발자**가 되고 싶었습니다. 직접 구현해보니 트랜잭션의 범위, 동시성 제어, 실패 처리 전략 등이 왜 중요한지 피부로 느낄 수 있었고, 앞으로 어떤 도구를 쓰더라도 더 잘 활용할 수 있는 자신감을 얻었습니다. 이 프로젝트를 통해 얻은 깊이 있는 이해는 새로운 기술을 도입하거나 기존 시스템을 개선할 때 큰 자산이 될 것이라고 생각합니다."
